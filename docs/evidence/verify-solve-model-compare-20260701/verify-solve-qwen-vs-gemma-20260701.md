# qwen3.5:9b vs gemma4:e4b — verify_solve-specific blind replay

**Date**: 2026-07-01 · **Hardware**: M1, 16 GB · **Backend**: Ollama (production path)
**Motivates**: Phase 0 diagnostic step of the verify_solve wrong-answer-rate improvement work.
**Artifacts**: [`replay_qwen_vs_gemma.py`](replay_qwen_vs_gemma.py) (reproduce) · [`pilot-results-20260701.json`](pilot-results-20260701.json) (n=35) · [`full-results-20260701.json`](full-results-20260701.json) (n=110)

## Question

`logs/verification-audit.jsonl` (252 challenges, 2026-06-28 through 07-01 — entirely the gemma4:e4b production period) shows `llm_extract`/`llm_reason` wrong-answer rates of 16.1% / 66.7%. The ADR-0069 model swap to gemma4:e4b was validated on general comment/distill quality, never on this narrow adversarial-decode-plus-exact-arithmetic task specifically. Is gemma a worse fit for verify_solve than qwen3.5:9b (still resident locally), or is the wrong-answer rate a prompt/guard-design gap common to any model?

## Method

Read-only, offline replay: `solve_challenge_result()` called directly on the 142 non-`code_parse` audit rows (code_parse rows are model-independent, so replaying them adds no signal), decoded from their stored `challenge_b64`. No network calls (`submit_verification`/`record_verification_audit` never invoked). Model switch via `OLLAMA_MODEL` env var + `ollama stop` between models (memory safety: qwen 6.6 GB + gemma 9.6 GB > 16 GB).

**Attribution join** (the reason a naive "replay all `verify_success=True` rows through both models" design would be wrong): `logs/llm-calls-*.jsonl` shows qwen3.5:9b only ever served `caller=moltbook.verify_solve` on 2026-06-28 before the gemma cutover that day (~12:39 UTC, matching ADR-0069); every row from 06-29 onward was generated by gemma. Comparing both models against the full "known correct" set would therefore be mostly gemma re-solving its own past answers — a self-consistency check dressed up as a fair fight. Each audit row's actual producing model was instead identified by recomputing `_challenge_prompt(challenge_text)`'s SHA256 (matching the telemetry's own `prompt_sha256[:12]`) and cross-referencing `num_predict` (512 → `llm_extract` tier, {3000, 5000} → `llm_reason` tier) with timestamp-proximity tie-breaking. This produced a clean 0-unattributed split: **95 gemma-solved rows**, **15 qwen-solved rows** (both `verify_success=True`).

Ran as a ~25/10-row pilot first (validate the join/env-switch/`ollama stop` mechanics), then the full 95/15-row set.

## Results

| Metric | Pilot (n=35) | Full (n=110) |
|---|---|---|
| **qwen blind-replay of gemma-solved (PRIMARY)** | 19/25 = **76.0%** | 69/95 = **72.6%** |
| qwen self-consistency floor | 10/10 = 100% | 15/15 = 100% |
| gemma blind-replay of qwen-solved (secondary, small n) | 10/10 = 100% | 15/15 = 100% |
| gemma self-consistency floor | 25/25 = 100% | 95/95 = 100% |

Both self-consistency floors sit at 100% in both runs — no sampling noise from either model at `temperature=0.0` on repeat runs. gemma reproduces qwen's small solved set perfectly too. qwen's blind replay of gemma's solves is the only number below 100%, consistent between the 25-row pilot (76%) and the 95-row full run (72.6%).

One qwen mismatch surfaced a real code guard gap (not a qwen-specific issue): a guarded-extraction call produced a self-consistent EXPR/FINAL pair of `"-100.00"`, which `_extract_guarded_answer` accepted outright because `_compute_decimal_pair` had no non-negativity check (unlike `code_parse_challenge`'s existing non-negative domain assumption). Fixed as part of this work regardless of the model question (see the paired improvement-plan commit).

## Verdict

**gemma4:e4b is not meaningfully worse than qwen3.5:9b at this task** — gemma's own accuracy on this corpus (100% self-consistency across 110 rows) is at least as good as qwen's blind-replay accuracy (72.6%), and there is no evidence a qwen swap would raise the ceiling. The ~27% qwen mismatch rate against gemma's answers reads as ordinary cross-model/prompt-sensitivity variance on a hard adversarial-decode task, not a gemma-specific weakness — if anything the data leans the other way. **Per-task model routing (pin verify_solve to qwen) is not warranted.** The wrong-answer rate is a prompt/guard design gap common to any model, so the code-level fixes (reasoning-path self-consistency guard, code_parse vocabulary/grammar extension) are the correct lever, independent of which model serves the call — matching the priority given to them in the accompanying implementation.
