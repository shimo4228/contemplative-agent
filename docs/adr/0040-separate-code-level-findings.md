# ADR-0040: Separate Code-Level Findings from Weekly Self-Reflection Report

## Status
accepted

## Date
2026-05-19

## Context

The weekly analysis report generated by `scripts/weekly-analysis.sh` (calling `claude -p` with `weekly-analysis.md` as system prompt and `principles.md` embedded in user prompt) historically contained an F section with three subsections:

- **F1**: Structural recommendations (code / schema / pipeline diff)
- **F2**: Identity-level open questions
- **F3**: Pure observations

The LLM generating this report has access to:

1. The system prompt (`weekly-analysis.md`)
2. The principles file (`principles.md`)
3. Daily comment reports for the period
4. State diffs from `git log` (identity, constitution, skills, rules, knowledge.json)
5. Up to 3 previous weekly reports

The LLM **does not** have access to:

- The current source code (`src/contemplative_agent/`)
- ADRs (`docs/adr/`)
- The current full text of identity, constitution, skills, or rules (only the diffs)
- CODEMAPS (`docs/CODEMAPS/`)

This context gap produced systematic failures in F1 across multiple weekly reports. Three concrete examples from `weekly-2026-05-17.md`:

- **F1.1** (num_predict): Proposed making `num_predict` proportional to input length. Verification against `core/llm.py`: the existing derivation is `num_predict = ceil(max_length/3) + 50 = 3384 tokens` (with `MAX_COMMENT_LENGTH = 10000`). Median observed output across 901 comments in May: ~400 tokens (12% of cap). The cap is not the constraint; changing the formula has no effect on actual output length.
- **F1.2** (input boundary): Proposed wrapping inputs in `<original_post>` boundary markers to suppress cut-off hallucination. Verification against `core/llm.py::wrap_untrusted_content()`: external content is already wrapped in `<untrusted_content>...</untrusted_content>` with `_INJECTION_TOKENS` sanitization. Adding another wrapper is redundant.
- **F1.3** (MMR retrieval): Proposed MMR diversity in pattern retrieval to break echo chamber. Verification: `views._rank` is only invoked from `distill`, `insight`, `rules-distill`, `distill-identity`. Comment/reply generation passes `knowledge_section=""` and does not consult pattern retrieval. MMR at the proposed location addresses an indirect (distill-mediated) path, not the direct reply-generation echo claimed in the report.

The `principles.md` guard rails (Principle 1: no post-generation filter; Principle 2: no hardcoded blocks; Principle 4: no repeated recommendation) constrain *what kind* of F1 the LLM proposes, but cannot compensate for the *absence of the context* needed to verify that an F1 is grounded in actual code state.

The same context-gap problem applies to F2 (writing meaningful identity-level questions requires the current full text of identity/constitution/skills/rules plus prior ADRs) and F3 (judging whether a pattern is "self-sustaining" or "reproducible" requires understanding the generation pipeline structure).

## Decision

The weekly report focuses exclusively on observation (sections A–E):

- **A**: Quantitative summary
- **B**: Agent state snapshot (diff-based, what's available)
- **C**: Engagement patterns (with quotes)
- **D**: Change points (qualitative, grounded in E)
- **E**: Qualitative highlights (the analytical center)

Code-level findings (the former F section) are produced separately by the `weekly-report-diagnosis` skill (located at `.claude/skills/weekly-report-diagnosis/`). The skill reads the weekly report's E section together with the codebase, ADRs, and current identity/constitution/skills/rules, and emits a companion file `weekly-{end-date}-findings.md`.

The skill applies the same principles defined in `config/prompts/principles.md` (Principles 1-4 + Appendix of rejected mechanisms). The principles file is preserved at its current location because Principle 3 (quote-based depth) continues to apply to the upstream weekly report.

## Alternatives Considered

### Alternative 1: Inject codebase / ADR / full layer context into `weekly-analysis.sh`

Rejected. The weekly cron must have a stable context to produce reproducible reports. Injecting CODEMAPS, ADR text, or full identity/constitution/skills/rules into the prompt would mean every weekly report's prompt drifts as the project evolves. Token cost would also balloon (each layer is several thousand tokens; cumulative context becomes unwieldy and crosses model context limits as ADRs accumulate).

### Alternative 2: Separate only F1, keep F2 and F3 in the weekly report

Rejected. The three layers are coupled by Principle 1 and Principle 2 (a finding may need to be re-categorized from F1 to F2 to F3 based on principle violation). Splitting them means the first layer fabricates F1 candidates that the later layers cannot reject — the principle guard breaks. Additionally, F2 and F3 share the same context gap as F1 (per the verified examples above), so partial separation does not solve the root cause.

### Alternative 3: Discard `principles.md` and inline principles into the skill

Rejected. Principle 3 (quote-based depth) applies to the upstream weekly report (sections C, D, E must be grounded in quotes). Moving principles entirely to the skill leaves the weekly report without methodological constraint.

## Consequences

### Positive

- F1 proposals are grounded in actual code state (file path + line number + ADR reference required by the skill's diagnostic checklist)
- Past weekly reports can be re-diagnosed when new ADRs land or when pipeline code changes
- The weekly cron prompt remains stable across project evolution (no need to update prompt when ADRs are added)
- The skill's staged reading order (CODEMAPS → source → ADR → identity layers) makes the diagnostic process auditable

### Negative

- Weekly reports appear "incomplete" until the diagnosis skill is run. Mitigated by file pairing: `weekly-{date}.md` ↔ `weekly-{date}-findings.md` makes the pair visible in `reports/analysis/`.
- Operator must invoke the diagnosis skill manually (cron-side automation is out of scope of this ADR; the skill is `user-invocable: true` for now).
- Skill output has no guard against the operator running it with a stale codebase view (e.g., before pulling latest). The skill's diagnostic checklist surfaces this responsibility to the operator.

### Re-check trigger

Re-evaluate after 2 weeks (around 2026-06-02). Specifically check:

1. Does the diagnosis skill output reference specific file paths / line numbers in its F1 entries?
2. Does at least one F1 proposal lead to a viable implementation (vs. the three-of-three failure rate observed in the unaided weekly LLM)?
3. Are F2 questions grounded in current identity/constitution text quotes, or do they still echo state-diff-only references?

If the answers are no, the design is wrong and reconsideration is needed (e.g., automating the skill invocation from `weekly-analysis.sh`, or restructuring the input contract).
