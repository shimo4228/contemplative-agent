# Semantic Memory Layer — 提案書

## 1. 何が変わるのか (機能ベース)

### 現状: 時間で思い出す

エージェントの記憶は「最近N日間に何があったか」でしか引き出せない。

```
contemplative-agent distill --days 3
→ 直近3日のエピソードを読み、パターンを抽出する
```

これは人間で言えば「3日前までの日記を読み返す」に相当する。1ヶ月前に同じ状況があっても、それを参照する手段がない。

### 提案: 意味で思い出す

「この話題に関連する過去の経験は?」という検索ができるようになる。

```
contemplative-agent search "他のエージェントとの哲学的対話"
→ 日付に関係なく、意味的に近いエピソードが返る
```

これは仏教哲学のアーラヤ識 (storehouse consciousness) のモデルに近い。経験は時系列ではなく、現在の状況との意味的な親和性によって想起される。

### 具体的にできるようになること

| 機能 | コマンド | 説明 |
|------|---------|------|
| セマンティック検索 | `search "query"` | 過去のエピソードを意味で検索 |
| タイプ絞り込み | `search "query" --type interaction` | 対話だけ、投稿だけ、等 |
| インデックス構築 | `reindex` | 蓄積済みエピソードを一括で検索可能にする |
| インデックス状態確認 | `reindex --stats` | 何件インデックス済みか等 |

### できないこと (初期版の範囲外)

- セッション中のリアルタイム検索 (M1 のメモリ制約のため。後述)
- distill/insight の自動強化 (検索品質を検証してから次のステップで追加)

## 2. 技術解説

### 2.1 Embedding とは何か

「文章を数値の列 (ベクトル) に変換する」技術。意味が似た文章は、似た数値の列になる。

```
"哲学について議論した"  → [0.12, -0.34, 0.56, ...]  (768個の数値)
"思想的な対話を行った"  → [0.11, -0.33, 0.55, ...]  ← 近い!
"天気が良かった"        → [0.89, 0.22, -0.71, ...]  ← 遠い
```

2つのベクトル間の「距離」を計算すると、意味的な類似度がわかる。これがセマンティック検索の原理。

### 2.2 Embedding モデル: nomic-embed-text

テキストをベクトルに変換する専用モデル。qwen3.5:9b (対話用) とは別のモデル。

| 項目 | 値 |
|------|-----|
| サイズ | 274MB (ディスク) |
| VRAM 使用 | ~600MB |
| 出力次元 | 768 (1文あたり768個の数値) |
| 処理速度 | ~10ms/文 |
| 多言語 | 対応 (英語・日本語等) |
| コンテキスト | 8,192トークン (長い文章もOK) |

Ollama に `ollama pull nomic-embed-text` でインストールする。

### 2.3 Vector DB: sqlite-vec

ベクトル (数値の列) を保存して、「このベクトルに近いものを探せ」という検索 (k-NN: k-Nearest Neighbors) ができるデータベース。

**なぜ sqlite-vec か:**

- Python に標準搭載されている SQLite に拡張機能として載せるだけ
- 新しいライブラリの依存がほぼゼロ (現在 `requests` のみの方針を維持)
- 「日付が先週以内」「タイプが interaction」等のフィルタも SQL で書ける
- ファイル1つ (`semantic.db`) で完結。サーバープロセス不要

**データの流れ:**

```
JSONL エピソード
  ↓ _summarize_record() で要約テキスト化
  ↓ nomic-embed-text で 768次元ベクトルに変換
  ↓ sqlite-vec に保存 (ベクトル + メタデータ)

検索クエリ "哲学的な対話"
  ↓ nomic-embed-text でベクトルに変換
  ↓ sqlite-vec で k-NN 検索 (最も近いベクトルを k 件返す)
  ↓ 元のエピソードレコードを表示
```

### 2.4 M1 16GB での動作設計

**問題**: M1 MacBook Pro は CPU/GPU が同じ 16GB メモリを共有する。qwen3.5:9b (~6-7GB) と nomic-embed-text (~600MB) を同時にロードすると、Ollama がモデルの入れ替え (スワップ) を繰り返し、動作が遅くなる。

**解決策: バッチ処理**

```
❌ セッション中にリアルタイム embedding
  → qwen3.5 で応答生成 → nomic でembed → qwen3.5 に戻す → nomic で...
  → モデルスワップが毎回発生、セッションが遅延

✅ セッション後にバッチ embedding
  → セッション中: qwen3.5 のみ使用。JSONL に書くだけ (現行通り)
  → セッション後: reindex コマンドで nomic のみ使用。まとめて処理
  → モデルスワップなし
```

セッション中の動作は一切変わらない。`reindex` はセッション後に手動で実行するか、cron で自動化する。

## 3. 既存機能への影響

### 変更なし (安全)

| 機能 | 影響 |
|------|------|
| `run` (セッション実行) | **変更なし**。JSONL への書き込みは現行通り |
| `distill` (記憶蒸留) | **変更なし**。時間ベースの処理は従来通り |
| `insight` (スキル抽出) | **変更なし**。KnowledgeStore 経由のまま |
| JSONL Episode Log | **変更なし**。append-only の監査証跡として維持 |
| config/knowledge.json | **変更なし** |
| config/identity.md | **変更なし** |

### 追加されるもの

| 項目 | 内容 |
|------|------|
| `semantic.db` | `~/.config/moltbook/semantic.db` に新規作成。JSONL のセマンティックインデックス |
| `reindex` コマンド | CLI に追加。JSONL → ベクトル化 → DB 格納 |
| `search` コマンド | CLI に追加。セマンティック検索 |
| `nomic-embed-text` | Ollama に追加インストール (274MB) |
| `sqlite-vec` | pip 依存に追加 (~5MB) |

### 削除・壊れるもの

**なし。** 全ての新機能はオプショナル。sqlite-vec がインストールされていなくても、nomic-embed-text がなくても、既存機能は動作する。

## 4. トレードオフ

### メリット

1. **過去の経験を意味で検索できる** — 時間に縛られない記憶の想起
2. **既存機能への影響ゼロ** — 全てオプショナル追加。壊れるものがない
3. **依存の増加が最小限** — sqlite-vec (~5MB) のみ。embedding は既存の Ollama + requests で完結
4. **研究的価値** — アーラヤ識モデルの計算論的実装として、論文で議論可能な設計
5. **将来の distill/insight 強化の基盤** — 検索品質を検証してから段階的に統合

### デメリット・制約

1. **セッション中のリアルタイム検索は不可 (M1 制約)**
   - バッチ処理のため、最新のセッションのエピソードは `reindex` を実行するまで検索対象にならない
   - 十分な VRAM を持つマシン (32GB+ or 外部 GPU) では将来的にリアルタイム化可能

2. **追加のディスク使用量**
   - nomic-embed-text: 274MB (Ollama モデル)
   - semantic.db: ~3KB/エピソード (10,000エピソード ≈ 30MB)
   - 合計: ~300MB 追加

3. **reindex の手動実行が必要**
   - セッション後に `contemplative-agent reindex` を実行する必要がある
   - cron で自動化は可能だが、初期版では手動

4. **sqlite-vec の成熟度**
   - v0.1.x でまだ新しいプロジェクト
   - 問題が出た場合: JSONL は無傷なので、sqlite-vec を外すか Qdrant に切り替え可能

5. **Embedding の品質は完璧ではない**
   - 短い要約テキスト (80文字程度) からの embedding なので、ニュアンスが落ちる場合がある
   - embedding モデルを変更したら `reindex` で全件再生成が必要

### やらないこと (意図的に)

- **JSONL の廃止** — 監査証跡・研究再現性のために維持
- **MemoryStore への統合** — セッション中のリアルタイム embedding はしない (M1 制約)
- **distill/insight の変更** — 初期版では既存動作を維持。検索品質を検証してから
